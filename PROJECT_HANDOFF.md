# Передача проекта Ykamina.ru

Актуальность документа: 19 июля 2026 года.

## 1. Что это за проект

Ykamina.ru — интернет-магазин каминов и сопутствующих товаров. Проект включает публичный каталог, карточки товаров, избранное, корзину, оформление и оплату заказов, кабинеты покупателей и сотрудников, административную панель, аналитику продаж и импорт каталога поставщика RealFlame.

Сейчас приложение представляет собой единый Next.js-проект: страницы, серверные обработчики и работа с PostgreSQL находятся в `frontend/`. Папка `backend/` зарезервирована под будущий отдельный сервис, но пока не используется.

## 3. Технологии

- Next.js 14.2, React 18, TypeScript 5.8;
- Tailwind CSS 3.4;
- PostgreSQL 16, прямые SQL-запросы через `pg`;
- парсинг CSV через `csv-parse`, XLSX через `xlsx`;
- аутентификация на собственных сессиях в PostgreSQL;
- локальная разработка базы через Docker Compose;
- платежи: Т-Банк и Яндекс Pay / Сплит.

ORM, отдельного backend-сервиса, очереди задач и автоматического тестового набора пока нет.

## 4. Структура

- `frontend/src/app/` — страницы, server actions и API-обработчики;
- `frontend/src/components/` — интерфейс магазина, кабинетов и панелей управления;
- `frontend/src/lib/` — каталог, база, пользователи, корзина, заказы и платежные интеграции;
- `frontend/scripts/` — миграции, создание суперадминистратора и импортеры;
- `frontend/public/` — статические изображения и логотипы;
- `frontend/certs/timeweb-ca.crt` — сертификат для подключения к облачной инфраструктуре Timeweb;
- `database/migrations/` — 16 последовательных SQL-миграций;
- `database/seeds/` — демонстрационные данные;
- `docker-compose.yml` — локальный PostgreSQL;
- `docs/README.md` — краткие инструкции разработчика;
- `backend/` — заготовка под будущий отдельный сервис.

## 5. Возможности приложения

### Публичная часть

- главная страница с промо- и акционными блоками;
- каталог с категориями, фильтрами и сортировкой;
- карточка товара с галереей, характеристиками и совместимыми товарами;
- акции;
- избранное и сохранённые подборки в `localStorage`;
- корзина в `localStorage`;
- оформление заказа;
- страницы доставки, FAQ, правил работы и реквизитов;
- результат оплаты.

### Пользователи и роли

Роли: `customer`, `sales_manager`, `super_admin`.

- регистрация и вход по email/паролю;
- пароль хранится как `scrypt`-хеш с солью;
- сессия хранится в БД, браузеру выдаётся `HttpOnly`, `SameSite=Lax` cookie на 30 дней;
- покупатель видит личный кабинет;
- менеджер получает каталог, редактирование товаров, продажи и аналитику;
- суперадминистратор управляет каталогом, публикацией, пользователями, ролями, блокировкой, заказами, импортами и аудитом.

### Заказы и оплаты

- заказы хранят сумму, покупателя, позиции, источник, менеджера, поставщика платежа и человекочитаемый номер;
- поддержаны заказы с сайта и от менеджеров;
- Т-Банк: создание платежа, HTTPS-уведомления, онлайн-чек и полный возврат;
- Яндекс Pay / Сплит: создание заказа и проверка результата;
- частичный возврат пока не реализован;
- Яндекс Pay сейчас настроен на `sandbox`, а `NEXT_PUBLIC_YANDEX_SPLIT_ENABLED=true`;
- локальные `NEXT_PUBLIC_SITE_URL` и `NEXT_PUBLIC_API_URL` сейчас равны `http://localhost:3000`, то есть production URL в этом файле не настроен.

### Поставщик RealFlame

Импортеры загружают товары, остатки, цены, акции, характеристики, галереи, фотографии и совместимость очагов с порталами. Основная синхронизация запускается вручную командой `npm run supplier:realflame:sync`. Для production её нужно вынести в cron или другой планировщик.

## 6. База данных

Облачная PostgreSQL размещена в инфраструктуре Timeweb. На момент проверки подключение работает.

Текущий снимок:

- база: `default_db`;
- технический пользователь: `gen_user`;
- товаров: 3345;
- пользователей: 3;
- заказов: 14;
- запусков импорта: 8.

Основные сущности: `categories`, `products`, `suppliers`, `prices`, `import_runs`, `product_compatibilities`, `users`, `user_sessions`, `admin_audit_log`, `payment_orders`.

Схема развивается миграциями `001`–`016`. Перед запуском новой копии необходимо выполнить все миграции через `npm run db:migrate`.

Важно: в локальной конфигурации установлено `DATABASE_SSL_REJECT_UNAUTHORIZED=false`. Это снижает строгость проверки TLS-сертификата; перед production следует настроить доверенный CA и включить полноценную проверку сертификата.

## 7. Конфигурация и секреты

Рабочие значения находятся только в `frontend/.env.local`. Этот файл исключён из Git и не должен отправляться в репозиторий, чат, задачу или обычную почту.

Сейчас в локальном файле заполнены:

- `DATABASE_URL` — строка подключения к PostgreSQL;
- `DATABASE_PUBLIC_HOST` — внешний хост облачной БД;
- `DATABASE_SSL_REJECT_UNAUTHORIZED` — режим проверки TLS;
- `NEXT_PUBLIC_API_URL` — адрес API/сайта;
- `NEXT_PUBLIC_SITE_URL` — публичный адрес сайта и база redirect/webhook URL;
- `REALFLAME_CSV_URL`, `REALFLAME_STOCK_URL` — источники поставщика;
- `TBANK_TERMINAL_KEY`, `TBANK_PASSWORD`, `TBANK_API_URL` — терминал Т-Банка;
- `TBANK_TAXATION`, `TBANK_TAX`, `TBANK_FFD_VERSION`, `TBANK_PAYMENT_METHOD` — параметры чека;
- `YANDEX_PAY_MERCHANT_ID`, `YANDEX_PAY_API_KEY`, `YANDEX_PAY_ENV` — Яндекс Pay;
- `NEXT_PUBLIC_YANDEX_SPLIT_ENABLED` — переключатель Сплита.

Дополнительно код поддерживает:

- `SUPER_ADMIN_EMAIL`, `SUPER_ADMIN_PASSWORD`, `SUPER_ADMIN_NAME`, `SUPER_ADMIN_PHONE` — создание администратора;
- `REALFLAME_SALES_URL` и переменные лимитов, задержек, повторов и параллельности импортеров;
- `NEXT_DIST_DIR` — отдельная директория сборки;
- `NODE_ENV` — режим приложения.

Секреты нужно передавать новому владельцу через менеджер паролей или защищённое хранилище. Предпочтительный вариант — не передавать старые пароли, а создать отдельные учётные данные новому владельцу и затем отозвать старые.

## 9. Безопасный порядок передачи

1. Новый владелец создаёт собственные аккаунты GitHub, Timeweb, Т-Банк и Яндекс.
2. Текущий владелец добавляет его по роли администратора/владельца, не передавая личные пароли.
3. Новый владелец подтверждает вход, двухфакторную аутентификацию и резервные коды.
4. Для базы, Т-Банка и Яндекс Pay выпускаются отдельные ключи/пароли.
5. Новые значения заносятся в защищённые переменные production-хостинга и локальный `.env.local` нового владельца.
6. Создаётся отдельный пользователь сайта с ролью `super_admin`.
7. Проверяются каталог, вход, создание заказа, тестовая оплата, webhook, возврат и импорт поставщика.
8. Только после проверки отзываются старые ключи, сессии и роли.
9. Отдельно передаются домен, DNS, биллинг, договоры эквайринга и почта.

## 10. Локальный запуск

Требуются Node.js 20+, npm, Docker и Docker Compose.

```bash
docker compose up -d postgres
cp frontend/.env.example frontend/.env.local
cd frontend
npm ci
npm run db:migrate
npm run dev
```

Сайт будет доступен на `http://localhost:3000`.

Для создания суперадминистратора задайте `SUPER_ADMIN_EMAIL`, `SUPER_ADMIN_PASSWORD` (не менее 12 символов), при необходимости имя и телефон, затем выполните:

```bash
cd frontend
npm run admin:create
```

Для одноразового случайного пароля можно использовать `npm run admin:create -- --generate`. Команда показывает пароль один раз.

## 12. Известные риски и незавершённые места

- Нет зафиксированной схемы production-деплоя и CI/CD.
- GitHub CLI требует повторного входа.
- Ветка «Долями» отправлена на GitHub, но ещё не объединена с `main`.
- Яндекс Pay работает в sandbox.
- Production URL не задан в локальной конфигурации.
- TLS-проверка сертификата БД ослаблена.
- Телефон на сайте выглядит демонстрационным.
- Нет автоматических тестов и отдельного мониторинга.
- Импорт RealFlame запускается вручную.
- Частичный возврат платежа не поддерживается.
- В README остались устаревшие пункты «следующих этапов», хотя часть из них уже реализована.

## 14. Что новый аккаунт не получит автоматически

Другой аккаунт ChatGPT/Codex не наследует историю этого чата, GitHub-сессию, доступ к кабинетам, локальные файлы или секреты. Если он работает на этом же компьютере и открывает эту же папку, репозиторий и локальный `.env.local` останутся на диске. Если работа переносится на другое устройство или в облачную среду, нужно отдельно:

- дать доступ к GitHub и клонировать репозиторий;
- безопасно создать новый `.env.local`;
- передать роли в сервисах из таблицы выше;
- предоставить этот документ;
- проверить рабочую ветку и незакоммиченные изменения.
